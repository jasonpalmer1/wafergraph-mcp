# wafergraph-mcp ‚Äî Remote MCP server for wafergraph.com's dataset

Read-only Cloudflare Workers MCP server exposing wafergraph.com's semiconductor & AI
supply-chain dataset (565 companies, 12 segments, 74 M&A deals, supplier/customer graph) as 5
tools for AI agents. No auth (v1, public data). Streamable HTTP transport at `/mcp`, human
landing page at `/`. Independent project ‚Äî not an official wafergraph product, but built to be
a good-faith front door to it (every response links back to wafergraph.com and its paid report).

Live: **https://wafergraph-mcp.jwpalm99.workers.dev**

## File map

- `src/index.ts` ‚Äî Worker entry point. Routes `GET /` ‚Üí landing page, `/mcp*` ‚Üí the MCP agent, else 404.
- `src/mcp-agent.ts` ‚Äî `WafergraphMCP extends McpAgent`; registers the 5 tools in `init()`.
- `src/data.ts` ‚Äî data layer: live-fetch + cache for companies/deals, vendored-snapshot read for taxonomy. See "Session memory" below for why it's split that way.
- `src/graph.ts` ‚Äî supplier/customer edge graph + `walkChain` (tiered BFS up/down, capped depth 2). Reimplemented cleanly from the *algorithm* in wafergraph's `site/src/data.js` (`buildChain`/`suppliersOf`/`customersOf`) ‚Äî not imported, per the read-only boundary on that repo.
- `src/types.ts` ‚Äî raw upstream shapes (`Company`, `Taxonomy`, `Deal`) + `AllowedCompany`/`toAllowedCompany()`, the single whitelist point that drops `key_products` (see field-discipline note below).
- `src/attribution.ts` ‚Äî fixed `attribution`/`links` blocks attached to every tool response.
- `src/landing.ts` ‚Äî the `/` HTML page (install snippets computed from the live request origin).
- `data/taxonomy.snapshot.json` ‚Äî vendored copy of wafergraph's `data/taxonomy.json`. Re-sync via `scripts/refresh-data.sh`.
- `scripts/refresh-data.sh` ‚Äî re-copies `taxonomy.snapshot.json` from `~/projects/wafergraph/data/taxonomy.json` (override source with `WAFERGRAPH_REPO`). Only file that needs this ‚Äî companies/deals are live-fetched.
- `wrangler.jsonc` ‚Äî Worker config: DO binding `WAFERGRAPH_MCP` ‚Üí class `WafergraphMCP`, `new_sqlite_classes` migration, `nodejs_compat`.
- `worker-configuration.d.ts` ‚Äî generated by `npx wrangler types` (gitignored; regenerate after touching `wrangler.jsonc`). Fully self-contained runtime ambient types ‚Äî no `@cloudflare/workers-types` dependency needed.

## Architecture

Cloudflare Worker (TypeScript) using the `agents` package's `McpAgent` pattern: a Durable
Object (SQLite storage backend, free on the Workers Free plan ‚Äî verified against current
Cloudflare docs before building, not assumed) handles the Streamable HTTP MCP session per the
current recommended Cloudflare pattern. `McpServer.registerTool()` (not the deprecated
`.tool()`) registers each tool with a Zod raw-shape `inputSchema`.

No persistent state is actually used ‚Äî every tool is a pure read over data fetched/cached in
`src/data.ts` ‚Äî so `WafergraphMCP`'s `State` type is an empty object; the DO exists only because
`McpAgent` requires one per session.

CORS: `McpAgent.serve()` defaults to permissive CORS (`origin: *`, exposes `mcp-session-id`,
allows the standard MCP methods/headers) ‚Äî verified in the installed package's own type/doc
comments, not configured by hand.

## Session memory

- **Data-source mode ‚Äî HYBRID, decided from real E2E testing, not the initial plan** (built
  2026-07-19): the brief's default assumption was pure live-fetch if the JSON is publicly
  reachable. `companies.json` and `deals.json` genuinely are ‚Äî confirmed via `curl` (`content-type:
  application/json`, `Access-Control-Allow-Origin: *`, upstream `Cache-Control: public,
  max-age=14400`) AND by this server's own successful live tool calls. **`taxonomy.json` is
  NOT** ‚Äî it 200s but serves the SPA's `index.html` shell (`content-type: text/html`), because
  wafergraph's `site/src/data.js` bundles taxonomy into the JS at build time and (unlike
  companies/deals) never emits it as a standalone static file ‚Äî confirmed both by curl and by
  reading that file's own comment (`// tiny (~5KB) ‚Äî stays bundled`). This was caught by the
  E2E verification step, not the earlier `curl -sI` recon pass (which only checked status codes
  and missed the content-type mismatch) ‚Äî worth remembering: **status 200 alone doesn't confirm
  a JSON endpoint is real; check content-type/body too.** Fix: `data/taxonomy.snapshot.json` is
  vendored (copied from `~/projects/wafergraph/data/taxonomy.json`, snapshot date 2026-06-23,
  surfaced in every `get_segments` response as `taxonomy_snapshot_date`), with
  `scripts/refresh-data.sh` to re-sync it. `src/data.ts`'s `fetchJSON()` also now throws loudly
  on any non-JSON content-type, so a repeat of this upstream-shape surprise fails fast instead
  of silently serving HTML as data.
- **Field discipline:** `key_products` is excluded from every response (`toAllowedCompany()` in
  `src/types.ts` is a whitelist, not a blacklist, specifically so this stays excluded by
  construction). Per wafergraph's own `CLAUDE.md`: that field was bulk-drafted with a known
  fabrication/mis-scope rate and a verified deep-fill is still in progress there. Re-enable by
  adding `key_products` back to `AllowedCompany`/`toAllowedCompany()` once that lands ‚Äî don't do
  it before, and don't re-derive/guess values here in the meantime.
- **Dependency security note:** the initial `npm install` (agents@0.2.10, @modelcontextprotocol/sdk@^1.15) resolved to versions with 2 high-severity advisories in the SDK copy nested inside `agents` (cross-client data leak via shared transport reuse, DNS rebinding not on by default, a ReDoS) ‚Äî real risk for a public remote server. Upgraded to `agents@^0.17.4` (which pulls a patched nested SDK) + `@modelcontextprotocol/sdk@^1.29.0` top-level; `agents@0.17.4` peer-requires `zod@^4`, so zod was bumped from the originally-planned v3 to v4 (no code changes needed ‚Äî the basic schema surface used here, `z.string()/z.number()/z.enum()/.optional()/.default()`, is unchanged between v3 and v4). `npm audit` is clean post-upgrade.
- **This repo is local-only by design** (task boundary) ‚Äî `git init` + commits happened, but nothing was pushed to GitHub. That's Jason's call, not automatic here, unlike wafergraph's own auto-push standing permission (which is scoped to that repo only anyway).
- **`~/projects/wafergraph` was read-only for this build** ‚Äî CLAUDE.md and `data/*.json`/`data/enrichment/` were read for reference/vendoring (just the one taxonomy.json copy), nothing there was written, git-touched, or branched. A concurrent session was/is actively working in that repo during this build.
